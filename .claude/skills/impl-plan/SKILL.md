# /impl-plan コマンド

`$ARGUMENTS` の実装計画を作成します。

## 準備ステップ

コードを書く前にコンテキストを固めることで、後戻りのコストを下げられる。以下を順に行う：

1. `.claude/specs/summary.md`（仕様）と `.claude/tech/summary.md`（技術）を読んで関連ドキュメントを特定する
2. 対象機能に関連するコードファイルを読んで現状を把握する
3. 不明点は `AskUserQuestion` でユーザーに確認してから計画を作る

## 出力先

`plan/[機能名]/` ディレクトリ配下に作成する：

- 単純なタスク（フェーズ分割不要）: `summary.md` 1本
- 複数フェーズ: `summary.md` + `01-[フェーズ名].md`, `02-[フェーズ名].md`, ...

機能名はケバブケース（例: `history-detail`, `suggestion-logic`）。

---

## summary.md の構成

summary は「このプランで何を・なぜ作るか」を一目で伝える地図。実装セッションをまたいでも迷わないように書く。

### 必須セクション

| セクション | 書くべき内容 |
| --- | --- |
| `## 目的` | 何を・なぜ作るか。完成像を一文で表す |
| `## 参照ドキュメント` | 実装時に必ず読む `.claude/specs/`・`.claude/tech/` のファイルをテーブルで列挙。「確認観点」列を付ける |
| `## フェーズ一覧` | `#` / フェーズ名 / 状態（未着手・進行中・完了） / 詳細ファイル名 のテーブル |
| `## 完了条件` | このプランが「終わった」と言える条件を箇条書きまたはチェックリストで列挙 |

### 任意セクション（状況に応じて追加、不要なら省略）

| セクション | 追加するタイミング |
| --- | --- |
| `## 現在地` | 既存機能の継続実装で、どこまで進んでいるかが重要なとき |
| `## 設計方針` | DB・API・画面設計など、実装前に合意が必要な選択肢があるとき |
| `## スコープ外` | 「やらないこと」が紛らわしく混入しやすい場合に明示 |
| `## 後続候補` | このプラン完了後に続く可能性が高い作業がある場合 |

---

## フェーズ詳細ファイルの構成

フェーズファイルは「このファイルだけ読めば実装できる」状態を目指す。曖昧な手順は実装中の判断コストを増やすため、具体性を最優先にする。

### 必須セクション

| セクション | 書くべき内容 |
| --- | --- |
| `## 目的` | このフェーズで達成することを一文で |
| `## 対象範囲` | 「今回やること」「今回やらないこと」を箇条書きで明示 |
| `## 変更対象ファイル` | 新規・変更ファイルを列挙（パスを明記） |
| `## 実装ステップ` | 番号付きの具体的な手順 |
| `## 完了条件` | チェックボックス（`- [ ]`）形式 |
| `## 検証方法` | 動作確認手順（実行コマンド or 手動操作ステップ） |

### 任意セクション

| セクション | 追加するタイミング |
| --- | --- |
| `## 状態管理` | Zustand store の追加・変更が含まれる場合 |
| `## この計画の位置づけ` | summary との関係が分かりづらい場合 |

---

## 詳細粒度の基準

粒度が粗いと実装中に判断が増え、計画の価値が下がる。以下を守る：

- 1ステップ = 「1ファイルへの変更または1つの確認操作」
- 「いくつかの変更を加える」のような曖昧な書き方は禁止。何をどのファイルに書くかを具体的に書く
- コマンド実行（`npx tsc --noEmit`, `npx expo lint` など）も1ステップとして明記する
- 実装ステップは上から順に実行すれば完了するように並べる

---

## 計画完成後

1. `summary.md` をユーザーに提示してレビューを依頼する
2. 承認を受けてからフェーズ詳細ファイルを作成する（または一緒に作る場合はその旨を確認する）
3. フェーズごとにサブエージェント委任の可否を判断してユーザーに提示する（下記参照）

---

## サブエージェント委任の判断

計画が固まった段階で、フェーズごとに以下の基準で委任可否を判断する。委任できるフェーズは「委任可能」と明示してユーザーに選択肢を提示すること。

### 委任に向くフェーズ

- 実装ステップが具体的で、途中の仕様判断が不要
- 対象ファイルと変更内容が明確に決まっている
- ユーザーとのやり取りなしに完結できる
- 他フェーズへの依存が少なく、独立して実行できる

### 委任に向かないフェーズ

- 実装中に設計の選択肢が出てくる可能性がある
- 複数ファイルの依存関係を逐次確認しながら進める必要がある
- DBスキーマ変更など、誤りの手戻りコストが高い
- ユーザーのフィードバックループが必要な UI 調整
