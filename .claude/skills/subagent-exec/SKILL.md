# サブエージェント実行

計画のタスクをサブエージェントに順次実行させる。メインのコンテキストを温存しながら実装を進めるのが目的。

## フロー

タスクごとに以下を繰り返す：

1. 計画ファイルからタスクを抽出し、必要なコンテキストをすべてまとめる
2. 実装サブエージェントに委任する
3. 完了後、仕様適合を1回レビューする
4. 問題があれば1回だけ修正させる（再レビューなし）
5. 次のタスクへ

## サブエージェントへの指示に含めること

実装サブエージェントはこのセッションのコンテキストを引き継がない。以下を必ず含める：

- タスクの目的と完了条件
- 変更対象ファイルのパス
- 実装ステップ（計画ファイルの該当箇所をそのまま渡す）
- プロジェクト固有の制約（型チェックと Lint を最後に実行すること。具体コマンドは `package.json` 確定後に CLAUDE.md と合わせる）

## 仕様レビュー

実装後にサブエージェントで仕様適合を1回確認する：

- 計画の完了条件をすべて満たしているか
- 変更対象外のファイルを触っていないか
- 型エラー・Lintエラーがないか

問題があれば実装サブエージェントに1回だけ修正させる。修正後の再レビューはしない。

## モデル選択

### 実装サブエージェント

タスクの複雑さに応じてサブエージェントのモデルを選ぶ。基本は opus-4.5 と opus-4.6 の2択。

| 状況                                     | モデル            |
| ---------------------------------------- | ----------------- |
| 仕様が完全で1〜2ファイルの変更のみ       | `claude-opus-4-5` |
| 複数ファイルにまたがり統合上の懸念がある | `claude-opus-4-6` |
| 設計判断や広範なコードベース理解が必要   | `claude-opus-4-6` |

### レビューサブエージェント

完了条件と実装の複雑さで切り替える。

| 状況                                                                                            | モデル              |
| ----------------------------------------------------------------------------------------------- | ------------------- |
| 完了条件が grep / tsc / lint / スコープ確認 で機械的に検証可能、かつ実装が1〜数ファイルのスタイル置換などの単純変更 | `claude-haiku-4-5` |
| 計画の解釈余地が広い／振る舞いロジックを触る／構造変更（コンポーネントの分割・移動・API変更など）を含む                     | `claude-opus-4-5`   |

判断に迷ったら opus-4.5 を選ぶ。レビューはコスト最適化より見落とし回避を優先する。

## 注意

- 実装中に設計判断が必要になったらサブエージェントを止め、ユーザーに確認する
- タスク間に依存がある場合は必ず順番に実行する（並列実行しない）
- 全タスク完了後、変更内容をユーザーに簡潔に報告する
