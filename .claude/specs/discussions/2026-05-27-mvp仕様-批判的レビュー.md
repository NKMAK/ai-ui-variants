# MVP仕様 批判的レビュー

- **対象仕様**: [../2026-05-27-v1-ui-variant-preview-agent-mvp.md](../2026-05-27-v1-ui-variant-preview-agent-mvp.md)（v1 初版）
- **日付**: 2026-05-27
- **目的**: 実装に入る前に、設計の穴・リスク・矛盾を洗い出す。

> 前提: 仕様の「割り切り」自体は妥当な判断が多い。以下は**実装前に潰しておくべき論点**を重要度順に並べたもの。

---

## 🔴 Critical — 設計の根幹に関わる矛盾

### 1. worktreeのbase(commit基準) と main snapshot(未コミット込み) のズレ

仕様の最大の穴。

- `git worktree` はcommitを基準に作られる → AIが見るコード＝**最後のcommit時点**。
- 一方 base snapshot は「セッション開始時の main worktree のファイル内容」＝**未コミット変更込み**。
- worktreeで生成した patch（commit基準のdiff）を、未コミット変更込みの main に当てると **conflictする**。
- さらに、**AIが見るコードとユーザーが見ている画面が食い違う**（ユーザーは未コミット変更込みの画面を見ている）。

**対策**: 「作業フロー制限」で**完全寄り（対象ファイルがcleanであることを要求）を選べばこの矛盾は消える**。仕様はこれをUX安全の理由だけで選んでいるが、本質は**整合性の担保**。完全寄りなら対象ファイルがworktreeのcommit内容と一致するので、patchが素直に当たる。これを設計判断として明記すべき。

### 2. AIに unified diff を直接生成させる脆さ

unified diff は**行番号・context行が1行でもズレると当たらない**形式で、LLMが最も間違えやすい出力。

**対策**: AIには **search/replace ブロック**（または「変更後のファイル全文」）を返させ、**こちら側でdiff/patchを構築**する。worktreeにファイルを書く → `git diff` でpatch化する仕様なら、AIは「変更後のコード」を返すだけでよく、patch生成は決定論的になる。「unified diffで返す」をAIの責務から外すのが堅牢。

### 3. 「画面状態を維持できる」は Fast Refresh 成立時のみ

仕様の売り文句だが、HMRは**フルリロードに落ちると in-memory state（form入力など）が飛ぶ**。

- 同一ファイル内のJSX/props/className変更 → Fast Refresh維持される ✅
- import追加・hooks構造変更・別ファイル波及 → フルリロード → 状態消失 ❌

**対策**: 対応修正タイプ（text/label/className/size）とはおおむね一致するので**実は整合的**だが、「状態維持」は無条件ではないと明記し、variant生成時の制約（同一ファイル・構造を壊さない）と紐付ける。

## 🟠 High — 実装で必ず刺さる

### 4. エディタで対象ファイルを開いている時の衝突

仕様に言及ゼロだが致命的になりうる。main worktree を裏で書き換えると:

- VSCode等で「ファイルが外部変更されました」通知が出る。
- ユーザーがそのファイルを**編集中（未保存バッファあり）**だと、保存時に preview の書き換えと衝突 → どちらかが消える。

**対策**: 最低限「セッション中は対象ファイルを触らない」前提を完全寄り制約に含める。理想は対象ファイルのロック表示。

### 5. クリック対象の source 曖昧性（intrinsic vs コンポーネント vs 呼び出し元）

`<SaveButton>` をクリックしたとき、ユーザーが変えたいのは:

- `SaveButton.tsx` の中身（定義）なのか
- `UserForm.tsx` の `<SaveButton/>` 呼び出し箇所（props/配置）なのか

ビルド時注入の `data-ui-source` は**最終的にDOMを出した intrinsic 要素の定義位置**を指すので、呼び出し元を変えたいケースとズレる。`callerSource` がoptionalで入っているのはこのためだが、**どちらをAIに渡すかの判断ロジックが未定義**。

**対策**: MVPでは「定義位置を主、callerはcontextとして添える」と決め打ちでよい。UIで「定義 / 使用箇所」を選べると後で効く。

### 6. 「前へ/次へ」連打時の競合

各切替が「snapshot復元 → patch適用 → HMR待ち」の非同期処理。連打すると中間状態が見えたり、HMR完了前に次の書き換えが走って壊れる。

**対策**: サーバ側で**セッション単位のロック**と、preview適用の**逐次化（最新リクエストのみ反映 / debounce）**が必須。

## 🟡 Medium — 割り切ってよいが明記すべき

| 論点 | 内容 |
|---|---|
| **単一セッション前提** | main worktree は1つ。同時に複数UIをいじれない。MVPは1セッション排他と明記を。 |
| **worktree は本当に要るか** | MVPでビルド検証しないなら、worktreeにファイルを書かず一時ディレクトリ/メモリ上で「変更後コード→patch」生成でも足りる。worktreeが効くのは**AIがビルド・型チェックして検証する**フェーズ。MVPでは過剰の可能性。 |
| **demoのdata-ui-source手書き矛盾** | 「最小デモ」でソースに `data-ui-source` を手書きしているが、本番は**ビルド時注入でソースに残らない**設計。デモ用の割り切りと明記しないと、実装者が手書きを正としてしまう。 |
| **禁止ファイルのガード実装** | package.json/.env/auth等の変更禁止は、**patch検証ステップ**として実装が必要（AIの自己申告に頼らない）。touchedFiles をallowlist/denylistで機械的に弾く。 |
| **line:column のセッション跨ぎズレ** | patch適用でファイルが変わると `data-ui-source` の行番号が古くなる。1セッション内なら問題ないが、apply後に再クリックする前にHMR再注入が要る。 |

---

## まとめ：実装前に決めるべき3点

1. **完全寄り制約を「整合性のため」として採用**（論点1・4・5を一気に解消する）。
2. **AIの出力を unified diff → search/replace or 全文に変更**、patch化はこちら側で（論点2）。
3. **variant生成制約を「同一ファイル・構造を壊さない」に縛り**、Fast Refresh維持を担保（論点3）。
